# ScalingDecision

Represents a single scaling determination made by the pipeline’s autoscaling subsystem. It captures the stage under evaluation, the direction and magnitude of the recommended consumer adjustment, the metrics that triggered the decision, and historical context from prior scaling actions. Instances are typically produced by a scaling controller and consumed by stage coordinators to enact consumer count changes.

## API

### StageName
`public string StageName`

Identifies the pipeline stage to which this scaling decision applies. The value corresponds to the logical name assigned during pipeline topology definition.

### Direction
`public ScalingDirection Direction`

Indicates whether the decision is to scale up, scale down, or maintain the current consumer count. The `ScalingDirection` enum typically includes values such as `Up`, `Down`, and `None`.

### Reason
`public string Reason`

A human-readable explanation of why the scaling decision was made. This string is generated by the scaling controller and may reference specific thresholds, metric values, or rule evaluations that triggered the action.

### FromConsumers
`public int FromConsumers`

The consumer count at the moment the decision was evaluated, before any scaling action is applied. This value serves as the baseline for the recommended change.

### ToConsumers
`public int ToConsumers`

The target consumer count recommended by this decision. The difference between `FromConsumers` and `ToConsumers` represents the magnitude of the scaling action.

### BufferFillPercent
`public double BufferFillPercent`

The fill percentage of the stage’s input buffer at the time of evaluation, expressed as a value between 0.0 and 100.0. This metric is often a primary signal for scaling decisions.

### BackpressureFrequency
`public double BackpressureFrequency`

The observed frequency of backpressure events for the stage, typically measured over a recent sampling window. Higher values indicate the stage is struggling to keep pace with upstream producers.

### DecidedAt
`public DateTime DecidedAt`

The UTC timestamp at which the scaling controller finalized this decision. Used for ordering decisions and calculating cooldown periods.

### CurrentConsumers
`public int CurrentConsumers`

The actual consumer count at the moment the decision is being applied or inspected. This may differ from `FromConsumers` if other scaling actions have occurred between evaluation and application.

### LastDecision
`public ScalingDecision? LastDecision`

A reference to the immediately preceding scaling decision for the same stage, if one exists. Null when no prior decision has been recorded. Enables the controller to detect oscillation patterns and enforce cooldown constraints.

### LastScalingActionAt
`public DateTime LastScalingActionAt`

The UTC timestamp of the most recent scaling action that was actually executed for this stage. Used in conjunction with cooldown configuration to prevent excessive adjustments.

### ScaleUpCount
`public int ScaleUpCount`

A cumulative counter of scale-up actions performed for this stage over its lifetime. Incremented each time a scale-up decision is successfully enacted.

### ScaleDownCount
`public int ScaleDownCount`

A cumulative counter of scale-down actions performed for this stage over its lifetime. Incremented each time a scale-down decision is successfully enacted.

## Usage

### Example 1: Creating and Applying a Scale-Up Decision

```csharp
// A scaling controller evaluates a stage and determines consumer increase is needed.
var decision = new ScalingDecision
{
    StageName = "data-transform",
    Direction = ScalingDirection.Up,
    Reason = "Buffer fill exceeded 80% threshold for 3 consecutive samples",
    FromConsumers = 4,
    ToConsumers = 6,
    BufferFillPercent = 87.3,
    BackpressureFrequency = 0.15,
    DecidedAt = DateTime.UtcNow,
    CurrentConsumers = 4,
    LastDecision = previousDecision, // nullable reference to prior decision
    LastScalingActionAt = DateTime.UtcNow.AddMinutes(-5),
    ScaleUpCount = 3,
    ScaleDownCount = 1
};

// Stage coordinator applies the decision.
if (decision.Direction == ScalingDirection.Up)
{
    stage.ScaleConsumersTo(decision.ToConsumers);
    decision.ScaleUpCount++;
}
```

### Example 2: Evaluating Cooldown Using Historical Context

```csharp
// Prevent scaling if a recent action occurred within the cooldown window.
bool CanApplyDecision(ScalingDecision decision, TimeSpan cooldown)
{
    if (decision.LastDecision == null)
        return true; // No prior decision, always allow.

    var elapsedSinceLastAction = decision.DecidedAt - decision.LastScalingActionAt;
    if (elapsedSinceLastAction < cooldown)
        return false;

    // Detect oscillation: if last decision was opposite direction, extend cooldown.
    if (decision.Direction == ScalingDirection.Up &&
        decision.LastDecision.Direction == ScalingDirection.Down)
    {
        return elapsedSinceLastAction >= cooldown.Multiply(2);
    }

    return true;
}
```

## Notes

- **Nullability of `LastDecision`**: Consumers must guard against `LastDecision` being null, particularly on the first scaling evaluation for a stage. Attempting to dereference a null `LastDecision` will result in a `NullReferenceException`.
- **Timestamp ordering**: `DecidedAt` and `LastScalingActionAt` are independent values. `DecidedAt` records when the decision was computed; `LastScalingActionAt` records when a previous decision was physically enacted. They may diverge if decisions are queued or deferred.
- **Counter integrity**: `ScaleUpCount` and `ScaleDownCount` are not automatically incremented by the decision object itself. The calling code is responsible for updating these counters upon successful enactment to maintain accurate lifetime statistics.
- **Thread safety**: This type is a plain data object with no internal synchronization. When multiple threads concurrently read or mutate the same instance—particularly the cumulative counters or the `LastDecision` reference—external locking or immutable replacement patterns must be employed to avoid race conditions.
- **Metric consistency**: `BufferFillPercent` and `BackpressureFrequency` are point-in-time snapshots captured during evaluation. They are not live metrics and may be stale by the time the decision is applied. Downstream consumers should treat them as diagnostic context rather than actionable signals.
